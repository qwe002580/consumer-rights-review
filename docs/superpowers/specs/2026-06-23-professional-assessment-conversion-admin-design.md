# 专业评估、客户转化与后台效率设计

## 目标

升级消费纠纷评估的专业度和转化能力，同时提升手机端案件管理效率：

- 展示“达成全部诉求”和“取得实质处理结果”两组概率区间
- 减少 DeepSeek 模板化表达，强化对具体案情的引用和判断
- 区分客户可见结果与后台完整分析，避免直接公开全部处理方法
- 使用更明确、更醒目的微信咨询引导
- 为后台详情增加可靠的返回入口和内部案件摘要一键复制

## 成功概率口径

### 两组概率

1. **达成全部诉求概率**：实现客户填写目标的完整程度，例如获得全额退款、全部解除分期或达到其完整赔偿诉求。
2. **取得实质处理结果概率**：获得部分退款、减免、补偿、合同解除、服务调整或其他对客户有实际价值的处理结果。

两组结果均显示为百分比区间，不显示单点数字。页面必须说明：区间基于当前填写信息和材料情况生成，不是结果承诺；材料缺失或事实不明确时，区间应更宽。

### 混合计算

概率不得完全交给模型自由生成。系统使用混合评估：

- 程序根据付款凭证、合同、聊天记录、宣传承诺、履行情况、争议类型、处理阶段、平台结果、分期情况和主要障碍计算基础分。
- DeepSeek 以结构化字段识别与本案有关的有利因素、不利因素和决定性问题。
- 程序将结构化因素映射为受限调整值，再计算最终区间。
- 全部诉求概率不得高于实质处理结果概率。
- 所有区间必须限制在 `0%–95%`，上下界顺序正确，并保持足够宽度以反映不确定性。
- 缺少关键材料时降低置信度并扩大区间，不得用虚假的精确数字掩盖信息不足。

规则负责稳定性和边界，模型负责案情识别，不允许模型直接控制最终概率。

## 专业分析结构

DeepSeek 生成并保存一份内部完整分析，至少包含：

- 针对本案事实的专业概述
- 有利因素
- 不利因素
- 决定性问题
- 完整处理策略
- 分阶段行动计划
- 建议补充材料及其作用
- 沟通策略或沟通建议
- 人工复核优先级

提示词要求明确引用客户填写的金额、时间、付款方式、处理阶段、争议点和现有材料。禁止反复使用“从现有信息看”“现阶段更适合”等固定开头；禁止泛泛复述表单、堆砌法条、虚构事实或作出结果承诺。

模型输出解析失败、接口超时或未配置 DeepSeek 时，系统使用本地兜底分析和规则概率，客户提交不得失败。

## 客户与后台信息分层

### 客户结果

客户只看到：

- 两组概率区间及口径说明
- 与本案事实相关的核心判断
- 主要有利因素和风险
- 当前缺失的关键材料
- 第一步具体动作
- 后续阶段标题和方向
- 人工复核咨询入口
- 免责声明

后续阶段不得展示渠道顺序、完整投诉路径、可直接照抄的沟通话术或内部策略细节。

### 后台结果

登录后的案件详情保留：

- 客户提交的全部信息
- 两组概率区间及评分因素
- 完整专业分析
- 全部分阶段行动计划
- 完整沟通建议
- 风险标记、案件状态和后台备注

API 在案件保存后只向客户返回公开结果结构，完整内部结果仅保存到数据库并通过受登录保护的后台读取。不得依赖前端隐藏来保护内部内容。

旧案件可能没有新增字段。后台用兼容转换器读取旧结构；缺少概率时显示“尚未评估”，原有摘要、风险、步骤和沟通建议仍正常显示。

## 咨询转化区域

采用“延续评估型”文案：

- 标题：**你的初步结果已生成**
- 说明：**添加案件顾问，发送本次评估结果，进一步核对退款空间和处理重点。**
- 按钮：**添加顾问继续评估**

按钮使用高对比暖橙红色，与现有米白和深绿色界面形成明确视觉层级。桌面端在核心结果后和页面底部各放置一次；手机端增加底部醒目咨询条，并为页面内容增加安全间距，不能遮挡免责声明或交互控件。

## 后台返回与复制

### 返回案件列表

案件详情顶部提供“返回案件列表”链接，固定指向 `/cases`。不使用 `history.back()`，确保从企业微信、收藏页或新标签打开时仍能可靠返回。

### 一键复制内部案件摘要

详情页顶部提供“一键复制内部案件摘要”按钮。复制文本按以下顺序排版：

1. 案件标题、编号、提交时间、状态和风险标记
2. 客户姓名与联系方式
3. 交易信息和目标诉求
4. 情况概述、核心争议、已有材料和当前障碍
5. 两组概率区间及主要评分因素
6. 完整初步判断、有利因素、不利因素和决定性问题
7. 全部处理步骤、补充材料和沟通建议
8. 后台备注

文本使用清晰分节标题、项目符号和编号步骤，适合粘贴到企业微信内部沟通或案件记录中。复制功能仅存在于登录后的后台，不提供给客户页面。

优先使用 `navigator.clipboard.writeText`；不可用时使用兼容复制方式。成功显示“已复制内部案件摘要”，失败显示明确提示，不静默失败。

## 模块边界

- `lib/schema.ts`：定义内部分析、客户公开结果和概率区间结构，并兼容旧记录。
- `lib/probability.ts`：纯函数计算基础分、模型因素调整、区间限制和置信度。
- `lib/prompt.ts`：生成面向内部完整分析的专业提示词。
- `lib/analysis.ts`：调用模型、解析内部结果、应用规则概率及生成兜底分析。
- `lib/public-analysis.ts`：从内部分析生成客户可见结果，统一实施信息裁剪。
- `app/api/analyze/route.ts`：保存完整分析，仅返回公开结果。
- `components/analysis-report.tsx`：展示客户概率、有限行动方向和咨询入口。
- `lib/case-copy.ts`：生成后台内部案件摘要纯文本。
- `components/case-detail.tsx`：提供返回、复制、状态和完整分析操作。

## 错误处理与安全

- DeepSeek 失败时使用兜底分析，不中断案件保存。
- 概率计算对缺失字段、旧数据和未知枚举采用保守默认值。
- 公开结果生成函数使用明确的允许字段列表，不使用删除敏感字段的黑名单方式。
- 客户响应不得包含联系方式、内部完整步骤、沟通话术、后台备注或模型原始响应。
- 剪贴板失败不得影响详情页其他功能。
- 咨询链接未配置时不渲染无效按钮。

## 测试与验收

自动测试覆盖：

- 两组概率的顺序、范围、区间宽度和材料缺失影响
- 同一输入得到稳定概率
- 模型异常时的兜底分析和概率
- 提示词要求引用具体案情并禁止模板化表达
- 公开结果允许字段和敏感内容排除
- 旧案件分析兼容
- 客户页只详细展示第一步，后续步骤只显示方向
- 新咨询文案和按钮数量
- 后台返回链接固定指向 `/cases`
- 内部复制文本的字段完整性、排版和隐私用途标识
- 复制成功与失败反馈

人工验收覆盖：

- 桌面和手机客户结果页的概率可读性、咨询按钮对比度和底部间距
- 手机后台详情返回功能
- 手机与桌面剪贴板复制
- DeepSeek 实际输出的具体性和重复套话情况
- Zeabur 部署后公开接口不泄露内部分析

## 部署

完成测试、代码检查和生产构建后推送 `main`，由 Zeabur 自动部署。部署完成后使用不含真实客户信息的测试提交检查公开结果、后台完整结果、企微案件通知和复制排版；测试记录在后台标记为关闭。
